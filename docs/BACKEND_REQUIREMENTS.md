# Требования к бэкенду приложения (на основе BPMN CIAN_PROD_part1, CIAN_PROD_part2)

Документ описывает однозначные, полные и проверяемые требования к бэкенду платформы объявлений о недвижимости. Источники: `docs/CIAN_PROD_part1.bpmn`, `docs/CIAN_PROD_part2.bpmn`.

---

## 1. Роли и участники

| Роль        | Описание |
|------------|----------|
| **Продавец** | Пользователь, размещающий объявления и управляющий ими. |
| **Платформа** | Система (бэкенд), обеспечивающая проверку, размещение, продвижение, уведомления и хранение данных. |
| **Покупатель** | Пользователь, ищущий недвижимость и запрашивающий показы. |
| **Администратор** | Пользователь платформы с правами модерации и просмотра всех данных: просмотр пользователей и объявлений, изменение статуса объявлений (в т.ч. снятие с публикации), просмотр обращений и платежей. |

---

## 2. Хранилища данных (Data Store)

- **Объявления** — хранилище объявлений (состояние: черновик, активно, продлено, архивировано и т.д.).
- **База данных** — хранилище обращений/истории взаимодействий (запросы на показ, отказы, подтверждения).

Требования к данным и связям между сущностями должны быть определены в отдельной спецификации модели данных; в процессах используются только ссылки на эти хранилища.

---

## 3. Часть 1: Публикация объявления, продвижение, срок размещения и архивация

### 3.1 Вход в личный кабинет и создание объявления

| ID   | Требование | Критерий приёмки |
|------|------------|------------------|
| R1.1 | Бэкенд предоставляет возможность продавцу войти в личный кабинет (авторизация/идентификация продавца). | Есть механизм аутентификации; после входа доступен контекст «продавец». |
| R1.2 | Бэкенд предоставляет возможность продавцу заполнить и отправить данные объявления (создание/редактирование черновика). | Есть API (или эквивалент) для создания/обновления данных объявления от имени продавца; данные сохраняются (например, как черновик). |

### 3.2 Проверка и размещение объявления (Платформа)

| ID   | Требование | Критерий приёмки |
|------|------------|------------------|
| R1.3 | После отправки данных объявления платформа выполняет проверку и размещение объявления. | Реализована сервисная операция «Проверить и разместить объявление»: валидация данных и запись в хранилище «Объявления» с состоянием «размещено/активно». |
| R1.4 | После успешного размещения платформа оповещает продавца и предлагает платное продвижение (Топ/Премиум). | Реализована отправка уведомления продавцу с предложением платного продвижения. |

### 3.3 Уведомление продавца о размещении и выбор продвижения

| ID   | Требование | Критерий приёмки |
|------|------------|------------------|
| R1.5 | Продавец получает уведомление о том, что объявление размещено. | Есть канал доставки уведомления (например, push, email, личный кабинет); продавец может узнать о факте размещения. |
| R1.6 | Продавец может выбрать: опубликовать без продвижения или подключить платное продвижение (Топ/Премиум). | В ответ на уведомление система учитывает выбор: «без продвижения» или «с продвижением»; при выборе продвижения инициируется сценарий оплаты. |

### 3.4 Оплата продвижения (Топ/Премиум)

| ID   | Требование | Критерий приёмки |
|------|------------|------------------|
| R1.7 | При выборе платного продвижения продавец может инициировать оплату (Топ/Премиум). | Есть возможность инициировать платёж за продвижение от имени продавца для конкретного объявления. |
| R1.8 | Платформа обрабатывает результат оплаты (в BPMN — мок: 60% успех, 40% ошибка). | Реализована обработка результата оплаты с двумя исходами: успех и ошибка; в тестовой среде допускается мок с заданной вероятностью (например, 60/40). |
| R1.9 | При успешной оплате платформа активирует услугу продвижения для объявления. | После успешного платежа объявление переводится в состояние «с продвижением» (Топ/Премиум) в хранилище объявлений. |
| R1.10 | При успешной оплате платформа подтверждает продавцу запуск продвижения. | Продавцу отправляется уведомление о успешной активации продвижения. |
| R1.11 | При ошибке оплаты платформа сообщает продавцу об ошибке. | Продавцу отправляется уведомление об ошибке оплаты. |
| R1.12 | После уведомления об ошибке оплаты объявление считается опубликованным без продвижения. | Состояние объявления остаётся «опубликовано без продвижения»; сценарий завершается без активации Топ/Премиум. |

### 3.5 Истечение срока размещения (30 дней)

| ID   | Требование | Критерий приёмки |
|------|------------|------------------|
| R1.13 | Для каждого размещённого объявления платформа отслеживает срок размещения 30 дней. | Реализован таймер/задача по каждому объявлению: по истечении 30 дней с момента размещения генерируется событие «Срок размещения 30 дней истёк». |
| R1.14 | При истечении 30 дней платформа оповещает продавца о скорой архивации объявления. | Отправляется уведомление продавцу о предстоящей архивации. |

### 3.6 Подтверждение актуальности и решение продавца

| ID   | Требование | Критерий приёмки |
|------|------------|------------------|
| R1.15 | Продавец получает уведомление о скорой архивации и может подтвердить актуальность объявления (да/нет). | Есть способ получить уведомление и ответить: объявление актуально (продлить) или не актуально (архивировать). |
| R1.16 | Если продавец указывает «объявление не актуально»: платформа по запросу архивирует размещение и обновляет хранилище «Объявления». | Реализован запрос на архивацию; сервис «Архивировать размещение» переводит объявление в архивное состояние; процесс завершается сценарием «Объявление снято». |
| R1.17 | Если продавец указывает «объявление актуально»: платформа по запросу продлевает размещение и обновляет хранилище «Объявления». | Реализован запрос на продление; сервис «Продлить объявление» продлевает срок размещения (например, ещё на 30 дней) и обновляет данные в «Объявления». |
| R1.18 | После продления платформа уведомляет продавца о продлении публикации. | Отправляется уведомление продавцу о продлении. |
| R1.19 | После получения уведомления о продлении сценарий для продавца завершается состоянием «Публикация продлена». | Факт продления фиксируется; продавец получает подтверждение. |

## 4. Часть 2: Поиск, обращение, показы и закрытие объявления

### 4.1 Поиск недвижимости (Покупатель)

| ID   | Требование | Критерий приёмки |
|------|------------|------------------|
| R2.1 | Покупатель может инициировать поиск недвижимости по фильтрам. | Есть API поиска с параметрами (фильтрами); запрос от имени покупателя (или анонимный поиск, если это допускается). |
| R2.2 | Платформа формирует выдачу объявлений, включая объявления с продвижением (Топ/Премиум), и возвращает её покупателю. | Реализована сервисная операция «Показать выдачу (включая Топ)»: выборка из хранилища объявлений с учётом фильтров и правил ранжирования (в т.ч. продвинутые объявления). |

### 4.2 Обращение к продавцу (запрос на показ)

| ID   | Требование | Критерий приёмки |
|------|------------|------------------|
| R2.3 | Покупатель может выбрать объявление и инициировать обращение к продавцу (связаться с продавцом). | Есть API (или эквивалент) для создания обращения/запроса на показ по выбранному объявлению от имени покупателя. |
| R2.4 | Платформа сохраняет обращение в историю (хранилище «База данных»). | Каждое обращение записывается в хранилище с привязкой к объявлению, покупателю и времени. |
| R2.5 | После сохранения обращения платформа оповещает продавца. | Продавцу отправляется уведомление о новом обращении/запросе на показ. |

### 4.3 Согласование показа (Продавец)

| ID   | Требование | Критерий приёмки |
|------|------------|------------------|
| R2.6 | Продавец получает уведомление об обращении и может согласовать время показа (да/нет). | Есть способ получить уведомление и ответить: показ согласован (с возможностью указать время) или не согласован. |
| R2.7 | Если показ согласован: продавец подтверждает показ; платформа оповещает покупателя о назначенном показе. | Реализовано подтверждение показа; платформа отправляет покупателю уведомление с данными о назначенном показе (время и т.д.). |
| R2.8 | Если показ не согласован: продавец отказывает; платформа оповещает покупателя об отказе. | Реализован отказ; платформа отправляет покупателю уведомление об отказе в показе. |

### 4.4 Решение покупателя после показа

| ID   | Требование | Критерий приёмки |
|------|------------|------------------|
| R2.9 | Покупатель может отметить, что приехал на показ и принял решение: покупает или не покупает. | Есть способ зафиксировать посещение показа и решение (покупаю / не покупаю). |
| R2.10 | При решении «не покупаю» сценарий для покупателя завершается без дальнейших действий по данному обращению. | Состояние обращения обновляется; уведомления о закрытии объявления не инициируются. |
| R2.11 | При решении «покупаю» покупатель инициирует действие «Покупаю», после чего продавцу передаётся необходимость закрыть объявление. | Реализована связь между решением «покупаю» и запросом к продавцу на закрытие объявления (например, уведомление или задача «Закрыть объявление»). |

### 4.5 Закрытие объявления (Продавец и Платформа)

| ID   | Требование | Критерий приёмки |
|------|------------|------------------|
| R2.12 | Продавец может закрыть объявление (после решения покупателя «покупаю» или по собственной инициативе). | Есть API (или эквивалент) для закрытия объявления от имени продавца. |
| R2.13 | При закрытии объявления платформа убирает объявление из списка активных (обновляет хранилище «База данных»/«Объявления»). | Реализована сервисная операция «Убрать объявление из списка активных»: смена состояния объявления так, чтобы оно не попадало в выдачу поиска. |
| R2.14 | После снятия с публикации платформа оповещает продавца о закрытом объявлении. | Продавцу отправляется уведомление о том, что объявление закрыто. |
| R2.15 | Продавец получает уведомление о закрытом объявлении; сценарий для продавца завершается. | Факт уведомления фиксируется; процесс для данного объявления завершён. |

### 4.6 Уведомление покупателя об отказе

| ID   | Требование | Критерий приёмки |
|------|------------|------------------|
| R2.16 | При отказе продавца в показе покупатель получает уведомление об отказе; сценарий для покупателя по данному обращению завершается. | После отправки уведомления об отказе состояние обращения обновляется; покупатель получает сообщение. |

## 5. Сводка сценариев (покрытие BPMN)

### Часть 1

| Сценарий | Требования |
|----------|------------|
| Публикация без продвижения | R1.1–R1.6, ветка «Нет» по выбору продвижения. |
| Публикация с продвижением (успешная оплата) | R1.7–R1.10. |
| Публикация с продвижением (ошибка оплаты) | R1.8, R1.11, R1.12. |
| Истечение 30 дней → уведомление о архивации | R1.13, R1.14. |
| Продавец подтверждает актуальность → архивация | R1.15, R1.16. |
| Продавец подтверждает актуальность → продление | R1.15, R1.17–R1.19. |

### Часть 2

| Сценарий | Требования |
|----------|------------|
| Поиск и выдача (включая Топ) | R2.1, R2.2. |
| Обращение к продавцу и уведомление | R2.3–R2.5. |
| Согласование показа → подтверждение → уведомление покупателя | R2.6, R2.7. |
| Согласование показа → отказ → уведомление покупателя | R2.6, R2.8, R2.16. |
| Показ → не покупаю | R2.9, R2.10. |
| Показ → покупаю → закрытие объявления продавцом → снятие с публикации и уведомление продавца | R2.9, R2.11–R2.15. |

## 6. Нефункциональные и общие требования

| ID   | Требование | Критерий приёмки |
|------|------------|------------------|
| R0.1 | Все операции с объявлениями и обращениями выполняются в контексте идентифицированного пользователя (продавец/покупатель/администратор) или системы (платформа). | Используются механизмы аутентификации и авторизации; в логах/аудите при необходимости фиксируется субъект действия. |
| R0.2 | Уведомления доставляются через определённые каналы (например, push, email, личный кабинет); при недоставке допустима повторная отправка или фиксация ошибки. | Описаны каналы уведомлений и политика повторных попыток/ошибок. |
| R0.3 | Критичные операции (размещение, архивация, продление, снятие с публикации) отражаются в хранилищах данных так, чтобы состояние объявления было однозначно восстанавливаемо. | Модель данных и сценарии использования обеспечивают консистентность состояния объявлений и обращений. |
| R0.4 | Бэкенд поддерживает роль администратора и разграничивает доступ к операциям по ролям. | Пользователь с ролью «Администратор» может входить в систему и получать доступ только к эндпоинтам, разрешённым для администратора; операции модерации (просмотр всех пользователей, объявлений, обращений, платежей; снятие/архивация объявления от имени платформы) доступны только администратору. |
